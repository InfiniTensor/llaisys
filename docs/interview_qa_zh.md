# LLAISYS 复试讲解与高频追问整理

这份文档不是普通 FAQ，而是按照“项目写进简历以后，老师会怎么问”来整理的。建议你不要死背整篇，而是先记住每一节的主线，再根据老师的兴趣往下展开。

## 1. 你先怎么介绍这个项目

### 1.1 30 秒版本
我做的是一个简化版大模型推理系统 LLAISYS，后端主要是 C++，前端测试和交互用 Python。我在这个项目里完成了 CPU 算子优化、NVIDIA 后端支持、单用户聊天服务，以及新模型 TinyLlama 的接入。项目重点不只是“调 API”，而是把张量、算子、运行时、模型推理和聊天服务整条链路打通。

### 1.2 1 分钟版本
这个项目的目标是从底层理解大模型推理系统。我这次主要完成了 4 部分工作。第一部分是 CPU 侧，把 `linear`、`embedding`、`rms_norm`、`rope`、`self_attention` 等算子做了 OpenMP 并行优化。第二部分是 GPU 侧，补了 NVIDIA runtime 和 8 个推理关键算子的 CUDA 路径，并处理了设备内存拷贝、KV cache 写回、最终 token 读取这些细节。第三部分是聊天服务，做了采样、OpenAI 风格接口、流式返回和单用户串行控制。第四部分是模型扩展，新增了 `Llama` 路径并接入 `TinyLlama`。我最后还把 Qwen2 和 TinyLlama 都在当前平台上做了真实推理和聊天验证。

### 1.3 2 分钟版本
如果老师愿意听长一点，可以这样讲：

1. 项目背景
这个项目本质上是在做一个教学版大模型推理系统。它不是直接调用现成推理框架，而是要求自己去实现 Tensor、算子、Runtime、模型推理，再逐步加聊天服务和新模型支持。

2. 我的工作
我这次完成的是项目 1、2、3、6。项目 1 主要是 CPU 并行优化；项目 2 是 NVIDIA 后端；项目 3 是聊天服务；项目 6 是接入新模型 TinyLlama。

3. 我觉得最有价值的点
最有价值的不是单个功能，而是我把“张量和算子”一直串到了“模型生成”和“聊天接口”这一层。也就是说，从底层内存、算子调度，到上层 prompt、采样、SSE 流式返回，这整条链路我是自己看过、改过、验证过的。

4. 我实际遇到的工程问题
比如 GPU 上设备指针不能直接用主机侧 `memcpy`，CUDA stream 和 Python/PyTorch 混用时要处理同步问题，本地聊天 CLI 会被环境代理打成 `502`，TinyLlama 的 GPU 严格测试还遇到了 `bf16/f32` 验证口径不一致的问题。这些问题都不是“只会调包”能看到的。

## 3. 如果老师先问整体架构

### 3.1 这个系统的整体分层是什么？
推荐回答：
- 最底层是 Runtime，负责设备、内存、stream、同步、拷贝。
- 往上一层是 Tensor，负责 shape、dtype、stride、storage。
- 再上一层是算子，比如 `linear`、`rope`、`self_attention`。
- 再上层是模型推理，把算子按 Transformer 的结构串起来。
- 最上层是 Python 包装、测试脚本、聊天服务和 CLI。

### 3.2 为什么这样分层？
推荐回答：
因为这样职责清晰。设备相关逻辑放在 Runtime，数据结构逻辑放在 Tensor，数学计算放在算子，模型结构放在推理层，上层服务就只需要关心 prompt、生成和接口格式。分层清楚以后，CPU 和 GPU 后端也更容易复用同一套上层逻辑。

### 3.3 这个架构里你最想强调哪一层？
推荐回答：
我最想强调 Runtime 和模型推理层。Runtime 决定了 CPU/GPU 的统一抽象；模型推理层则是把算子、KV cache、增量解码、采样真正串起来的关键。

## 4. 如果老师问 Tensor 和 Runtime 基础

### 4.1 什么是 Tensor？
推荐回答：
Tensor 不只是“多维数组”，它还包含 shape、dtype、stride、storage、offset。真正决定它怎么解释内存的，不只是 shape，还有 stride 和 offset。

### 4.2 stride 是什么？
推荐回答：
stride 表示跨某一维前进一步时，底层内存要跳多少。它决定了切片、转置、view 是否能零拷贝实现。

老师可能追问：
shape 对了为什么还可能出错？

你可以答：
因为 shape 只是“你怎么看这块内存”，但 stride 才决定“这块内存实际上是怎么排的”。如果 stride 不兼容，强行 view 会把数据解释错。

### 4.3 Runtime 抽象解决了什么问题？
推荐回答：
它把 CPU 和 GPU 的差异隐藏在统一接口后面，比如分配内存、设备切换、stream、同步、拷贝。上层 Tensor、算子、模型代码就不需要分别写一套 CPU 逻辑和一套 GPU 逻辑。

### 4.4 为什么设备内存不能直接 `memcpy`？
推荐回答：
因为普通 `memcpy` 是面向主机地址空间的，设备指针必须走显式的 H2D、D2H、D2D 接口。如果直接主机侧 `memcpy` 一个 GPU 指针，轻则数据错，重则直接崩。

### 4.5 你在这个项目里真的遇到过这个问题吗？
推荐回答：
遇到过。模型权重加载和 KV cache 写回一开始如果沿用主机侧思维，GPU 路径就会出问题。后来我统一改成通过 runtime memcpy 完成。

## 5. 如果老师问项目 1：CPU 优化

### 5.1 你具体优化了什么？
推荐回答：
我重点优化的是 `linear`，另外把 `embedding`、`rms_norm`、`rope`、`swiglu`、`self_attention` 都做了按 token / row / head 的 OpenMP 并行化。

### 5.2 为什么最先优化 `linear`？
推荐回答：
因为 Transformer 里大头算力基本都在矩阵乘。只优化小算子，对整体推理提升有限；`linear` 快了，端到端速度才会明显变化。

### 5.3 你为什么用 OpenMP？
推荐回答：
因为课程目标是先利用现有多核 CPU，把易并行的循环快速展开。OpenMP 接入成本低，适合这类按行、按 token、按 head 的并行任务。

### 5.4 并行一定会更快吗？
推荐回答：
不一定。并行会带来线程创建、调度、缓存竞争等开销。如果算子太小，甚至可能更慢。所以我的思路是优先并行那些计算量大、循环结构清晰的部分，比如 `linear` 和注意力相关算子。

### 5.5 你怎么判断优化有没有效果？
推荐回答：
一方面看单算子 benchmark，比如 `linear`；另一方面看端到端推理耗时。只看单算子不够，因为真正使用时还会有 prompt 构造、KV cache、tokenizer、采样等其它开销。

## 6. 如果老师问项目 2：NVIDIA 后端

### 6.1 你在 GPU 端做了哪些事情？
推荐回答：
我做了两件事。第一件是 runtime，把设备数查询、设备切换、内存分配、stream、同步、拷贝这些能力补齐。第二件是算子，把推理链路需要的 8 个算子补上了 NVIDIA 路径。

### 6.2 哪 8 个算子？
推荐回答：
`add`、`argmax`、`embedding`、`linear`、`rms_norm`、`rope`、`self_attention`、`swiglu`。

### 6.3 为什么 `linear` 用 cuBLAS？
推荐回答：
因为矩阵乘是标准高性能操作，直接用 cuBLAS 更稳，也更符合工程实践。自己写 GEMM 内核不仅复杂，而且很难在短时间内达到成熟库的效果。

### 6.4 为什么有些算子还保留 host fallback？
推荐回答：
因为课程作业优先目标是“跑通且正确”。像 `argmax`、`self_attention` 先走 host fallback，可以快速验证整条推理链。后续如果继续优化，再把 fallback 换成真正的高性能 CUDA kernel。

### 6.5 你在 GPU 端踩过什么坑？
推荐回答：
- 设备内存不能直接主机侧 `memcpy`
- CUDA stream 与 Python/PyTorch 的默认 stream 混用容易出同步问题
- 结果从设备回主机时，如果没有同步，可能读到旧数据
- 测试口径如果一边是 `bf16` 一边是 `f32`，贪心 token 会提前分叉

### 6.6 你为什么把 NVIDIA 侧统一切到默认 stream？
推荐回答：
因为公开算子接口现在是同步调用语义，而 Python/PyTorch 这边也有自己的默认 stream 体系。如果我在底层偷偷用另一个自定义 stream，但上层又按默认 stream 的可见性去理解，就可能出现“数据明明算了但还看不到”的问题。对这个项目阶段来说，统一使用默认 stream 更稳。

### 6.7 这是不是说明你不会用多 stream？
推荐回答：
不是。这里的关键不是“会不会”，而是“在当前抽象里是否合适”。如果上层没有把 stream 生命周期和同步语义设计清楚，强行多 stream 反而更容易出错。先保证正确性和语义清晰，是更合理的工程选择。

## 7. 如果老师问算子实现

### 7.1 `embedding` 做的事是什么？
推荐回答：
`embedding` 本质上就是按 token id 从词表矩阵里取对应的行。输入是 token 索引，输出是每个 token 对应的向量。

### 7.2 `linear` 数学上在做什么？
推荐回答：
它在做矩阵乘加偏置，也就是 `Y = XW^T + b`。在 Transformer 里，Q/K/V 投影、MLP 的上投影和下投影，本质上都是线性层。

### 7.3 `rms_norm` 和 LayerNorm 有什么区别？
推荐回答：
RMSNorm 只利用均方根来做归一化，不减均值；LayerNorm 同时减均值再除标准差。很多现代大模型，比如 LLaMA、Qwen2，会用 RMSNorm。

### 7.4 RoPE 是什么？
推荐回答：
RoPE 是一种旋转位置编码。它不是直接把位置向量加到输入上，而是让 query 和 key 的相邻维度按角度旋转，把位置信息编码进注意力计算。

### 7.5 `self_attention` 的核心步骤是什么？
推荐回答：
先算 `QK^T` 得到分数，再做 causal mask 和 softmax，最后乘 `V` 得到上下文表示。如果是 GQA 或 MQA，还要注意 query 头和 KV 头数量不一样的问题。

### 7.6 什么是 GQA？
推荐回答：
GQA 是 Grouped Query Attention，query 头数可以比 key/value 头数多。像 TinyLlama 就是这种结构，所以注意力实现里不能简单假设 `nhead == nkvhead`。

### 7.7 你在注意力里怎么处理 KV cache？
推荐回答：
对新输入 token 先计算本轮的 K/V，然后写回历史缓存；注意力计算时再用“当前 Q”去看“历史 K/V + 当前 K/V”。这样后续生成时每步只需要增量计算一个 token。

## 8. 如果老师问模型推理链路

### 8.1 从输入 prompt 到输出 token，大致流程是什么？
推荐回答：
先通过 tokenizer 把 prompt 转成 token id，然后做 embedding。接着每层依次做 attention norm、QKV 投影、RoPE、写入 KV cache、self-attention、输出投影、残差、MLP norm、门控 MLP、残差。最后做最终 norm 和词表投影，得到 logits，再通过 argmax 或采样选下一个 token。

### 8.2 为什么生成时第一步要把整个 prompt 一次性送进去？
推荐回答：
因为模型需要先把 prompt 对应的 K/V 全部建立起来。只有这样，后面自回归生成时才能每一步只喂最后一个 token 做增量解码。

### 8.3 为什么后续每步只喂最后一个 token 就行？
推荐回答：
因为历史信息已经被保存进 KV cache 了。当前步只需要新的 query 去和历史 K/V 交互，不需要把整个历史再算一遍。

### 8.4 你在 Python 侧是怎么把 HF 权重映射到后端结构里的？
推荐回答：
我读取 safetensors 的 header，按权重名把它们映射到后端固定的权重槽位，比如 `in_embed`、`out_embed`、各层的 `attn_q_w`、`mlp_gate_w` 等。这样后端推理代码就不需要理解 Hugging Face 的高层对象结构，只认统一的权重布局。

### 8.5 为什么要把权重统一转成 `float32`？
推荐回答：
一方面实现更简单，CPU 和 GPU 可以复用同一套推理逻辑；另一方面更有利于先保证功能正确。后续如果追求性能，再考虑半精度或更低比特路径。

## 9. 如果老师问项目 3：聊天服务

### 9.1 为什么项目 3 不是简单调用 `generate()` 就完了？
推荐回答：
因为聊天不是单次推理，它要处理多轮消息、prompt 格式、采样、流式返回、服务端接口、并发控制这些工程问题。项目 3 的价值就在于把模型能力真正做成“可交互的服务”。

### 9.2 你为什么选择 OpenAI 风格接口？
推荐回答：
因为兼容性最好。很多前端、脚本和第三方工具都能直接对接 `chat/completions` 这套协议，后续做演示和扩展都更方便。

### 9.3 你的聊天服务有哪些接口？
推荐回答：
- `GET /health`
- `POST /v1/chat/completions`

### 9.4 非流式和流式你是怎么处理的？
推荐回答：
非流式就是等整段回复生成完，再一次性返回 JSON；流式则使用 SSE，不断返回 `data: ...` 形式的增量文本块，最后用 `[DONE]` 收尾。

### 9.5 为什么你选择 SSE 而不是 WebSocket？
推荐回答：
因为这个项目只需要“服务端不断往客户端推文本”，SSE 更轻量，和 OpenAI 风格接口也更接近，实现成本更低。

### 9.6 为什么聊天服务要加单全局锁？
推荐回答：
因为当前模型实例内部有状态，尤其是 KV cache。如果两个请求同时改同一个模型状态，请求之间会串线。课程要求本身也是单用户服务，所以单全局锁是合理选择。

### 9.7 你是怎么构造聊天 prompt 的？
推荐回答：
优先用 tokenizer 自带的 `apply_chat_template`，因为这样和模型官方格式最一致。如果模型没有这个接口，再退回到简单的 `System/User/Assistant` 文本模板。

### 9.8 为什么流式返回不能按 token 直接 decode？
推荐回答：
因为 BPE 分词里，一个自然语言片段可能要多个 token 才完整。如果每来一个 token 就直接展示，用户看到的中间态可能会乱码或者断裂。所以我在服务里按 token 增量推理，但对外按“新增文本”返回。

### 9.9 你的 CLI 做了什么？
推荐回答：
CLI 维护完整对话历史，支持 `/clear`、`/exit`，可以走非流式或流式请求。它本质上是最小可用的聊天前端。

### 9.10 你在聊天服务上踩过什么坑？
推荐回答：
最典型的是本地请求被环境代理错误转发成 `502`。这是一个非常工程化的问题，不是模型问题。后来我让 CLI 对 `localhost/127.0.0.1/::1` 自动绕过代理。

## 10. 如果老师问采样

### 10.1 为什么聊天时不能一直用 argmax？
推荐回答：
argmax 太贪心，结果容易重复、刻板。聊天更适合采样，这样回答自然度更高。

### 10.2 `temperature` 是什么？
推荐回答：
它控制分布的“尖锐程度”。越低越保守，越高越发散。

### 10.3 `top_k` 是什么？
推荐回答：
只保留概率最高的前 K 个 token，再从这 K 个里面采样。

### 10.4 `top_p` 是什么？
推荐回答：
不是固定保留 K 个，而是保留累计概率达到阈值 `p` 的最小 token 集合，再从里面采样。

### 10.5 这三个参数应该怎么一起理解？
推荐回答：
`temperature` 是重塑分布，`top_k` 和 `top_p` 是裁剪候选集合。一般先调低温度控制稳定性，再用 `top_k/top_p` 控制多样性范围。

### 10.6 你在代码里怎么区分贪心和采样？
推荐回答：
当 `top_k == 1`、`top_p == 1.0`、`temperature == 1.0` 时，我直接走贪心路径；否则走采样接口。这么做一方面省掉不必要的采样开销，另一方面便于和 Hugging Face 做严格 token 对齐测试。

## 11. 如果老师问项目 6：TinyLlama

### 11.1 你为什么要接 TinyLlama？
推荐回答：
因为课程要求支持新模型。TinyLlama 是标准 `llama` 架构，规模相对适中，比较适合在当前环境下完成接入和验证。

### 11.2 你接入新模型时改了哪些层？
推荐回答：
- C/C++ 后端新增 `Llama` API
- ctypes 绑定新增 `llama` 路径
- Python 模型封装新增 `Llama`
- 模型工厂按 `config.json` 自动识别
- 推理测试入口扩展为可同时支持 Qwen2 和 TinyLlama

### 11.3 Llama 和 Qwen2 的结构完全一样吗？
推荐回答：
不完全一样，但这个项目里它们足够相近，可以共享同一套 decoder-only 推理主路径。差异更多体现在配置、权重命名、注意力头配置、模型类型识别这些层面。

### 11.4 你怎么判断自己真的支持了 TinyLlama，而不是“凑巧能跑”？
推荐回答：
我做了三类验证。第一类是严格 token 对齐；第二类是真实生成；第三类是聊天服务接口验证。只有模型能被工厂正确识别、能正常推理、能进聊天服务，我才认为是真正支持了新模型。

### 11.5 TinyLlama 验证里最关键的坑是什么？
推荐回答：
GPU 严格测试第一次失败时，我先怀疑是后端算错了。后来通过分步对比发现：LLAISYS CPU、LLAISYS NVIDIA、HF float32 都能对齐，真正的问题是 HF 在 GPU 上默认跑 `bf16`。这说明做验证时一定要先搞清楚“比较口径是否一致”，不能一看到 mismatch 就直接怀疑算子实现。

## 12. 如果老师问测试与验证

### 12.1 你怎么验证自己的实现是对的？
推荐回答：
我分三层验证。第一层是 runtime/tensor/ops 单元测试；第二层是 `test_infer.py` 的模型推理一致性测试；第三层是聊天服务的真实 API 冒烟。

### 12.2 为什么不能只跑最终聊天效果？
推荐回答：
因为聊天能出字，不代表底层实现就是对的。你可能只是“看起来能跑”，但 token、logits、缓存、数据拷贝都可能有隐藏问题。所以我更看重分层验证。

### 12.3 你是怎么定位 TinyLlama 严格测试失败原因的？
推荐回答：
我不是一上来就改后端，而是先做分层对比：
- 先看 HF 和 LLAISYS 的完整 token 序列什么时候开始分叉
- 再比较前 8 个 token、前 32 个 token
- 再比较 `LLAISYS CPU` 和 `LLAISYS NVIDIA`
- 再把 HF 显式切成 `float32`
最后发现不是后端逻辑错，而是验证精度口径不一致。

### 12.4 你怎么验证聊天服务不是“偶然通了一次”？
推荐回答：
我同时做了 `TestClient` 本地路由验证、真实 `uvicorn` 进程验证、CLI 请求验证，还复现并修复了代理导致的 `502`。这样可以把“路由逻辑问题”和“环境问题”区分开。

## 13. 如果老师问你改代码时的取舍

### 13.1 为什么你优先保证正确性，而不是一开始就追求最高性能？

### 13.2 为什么有些地方你选择了更保守的实现？
推荐回答：
比如统一默认 stream、保留 host fallback、权重统一转 `float32`，这些都是偏保守的设计。它们牺牲了一部分极致性能，但换来了实现复杂度降低、调试更明确、验证更稳定。

### 13.3 如果老师说“这不够工业级”，你怎么回答？
